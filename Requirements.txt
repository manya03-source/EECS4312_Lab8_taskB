##1 Problem Understanding
System purpose:
The system manages reminders and event registrations while automatically resolving scheduling conflicts and maintaining deterministic behavior. It is designed to reduce cognitive load for busy users while also providing clear explanations for system decisions when required.

Inputs to the system:
-> Reminder creation requests (time and reminder name)
-> Reminder updates or snooze requests
-> Reminder cancellation requests
-> Event registration requests
-> Event cancellation requests
-> Event capacity values
-> User interaction commands
Outputs produced by the system:
-> Ordered reminder schedules
-> Registration confirmations
-> Waitlist promotion notifications
-> System explanations for automatic decisions
-> Error messages for invalid or duplicate requests

##2 Assumptions
A1: The system receives valid timestamps for reminders and registrations.
A2: Users interact with the system through a reliable interface that correctly sends their requests.
A3: Reminder names or event identifiers are unique enough to differentiate between reminders.
A4: Event capacity values are predefined and provided to the system before registrations occur.
A5: Users may frequently modify reminders (e.g., snooze or cancel them).
A6: The system must operate deterministically so that identical inputs always produce the same outputs.

##3 Functional Requirements
FR1: The system shall allow users to create reminders with a timestamp and description.
FR2: The system shall display reminders in a deterministic order when multiple reminders share the same timestamp.
FR3: The system shall automatically resolve reminder conflicts without requiring manual user intervention.
FR4: The system shall allow users to register for events and maintain a waitlist when the event capacity is full.
FR5: The system shall automatically promote users from the waitlist when an event slot becomes available.
FR6: The system shall provide explanations when reminders are skipped, moved, or when waitlist promotions occur.
